# Mesh 项目宪法

> 本文档定义 Mesh 项目的基本开发规则和约定，所有 AI 在处理本项目时必须遵守。

---

## 一、核心原则

### 1. 阅读并更新 Memory 文件 ⭐

**执行规则**：
- 开始任何工作前，必须先读取 `MEMORY.md`
- 每次重大变更后，必须更新 `MEMORY.md`
- 更新内容包括：功能开发、技术决策、Bug 修复、版本发布

### 2. 更新文档 📚

**文档位置**：
- `docs/README.md` - 文档导航
- `docs/01-快速入门.md` - 快速入门
- `docs/02-命令说明.md` - 命令完整说明
- `docs/03-Agent协议.md` - Agent 协议
- `docs/04-多AI协作流程.md` - 多 AI 协作流程

**更新规则**：
- 代码改了，文档必须同步改
- 新增命令，必须更新命令说明
- 修改协议，必须更新协议文档

### 3. 理解产品约束 🎯

**核心约束**（必须牢记）：
- ❌ 无用户注册系统
- ❌ 无 Email 验证
- ❌ 无服务器架构
- ❌ 不依赖云服务
- ✅ 极简优先
- ✅ 单文件可执行
- ✅ 兼容现有 Markdown Memory

### 4. 代码质量标准 💻

**编码规范**：
- 遵循 Go 官方代码规范
- 函数不超过 50 行
- 复杂度不超过 10
- 每个函数必须有注释
- 错误处理必须完整

**测试要求**：
- 核心功能必须有单元测试
- 覆盖率要求：工具函数 100%，核心逻辑 80%

### 5. 安全与隐私 🔐

**数据安全**：
- 敏感信息记录到 `key.md`
- `key.md` 不提交到 Git
- 数据存储在本地，不出设备

**同步安全**：
- 同步空间内不存储明文密码
- 需要加密的内容需提供加密方案

## 二、敏感信息记录

### 记录规则

当用户反馈敏感信息时，AI 必须自动记录到 `key.md`：

```bash
# 敏感信息类型
- API Key
- Token
- Access Key
- Secret
- 密码
```

### 记录模板

```markdown
### [服务名称]
**类型**: API Key / Token / AccessKey
**用途**: [说明]
**时间**: YYYY-MM-DD
**敏感信息**: \`\`\`[实际的key]\`\`\`
```

## 三、版本号管理

### 版本号规则

采用语义化版本：`MAJOR.MINOR.PATCH`

- `MAJOR` - 不兼容的 API 变更
- `MINOR` - 向后兼容的功能新增
- `PATCH` - 向后兼容的 Bug 修复

### 必须更新的位置

每次发布新版本时，必须更新以下 4 个位置：

1. `MEMORY.md` - 更新 `当前版本` 字段
2. `docs/01-快速入门.md` - 更新版本号
3. `go.mod` - 更新版本号（如果有）
4. `cmd/mesh/main.go` - 更新 `--version` 输出

### 更新流程

```bash
# 1. 更新所有版本号
# 2. 更新 MEMORY.md 记录变更
# 3. 运行测试
# 4. 打 tag
git tag v0.1.0
```

## 五、多 AI 协作规范

### 团队角色

| 角色 | 人数 | 职责 |
|-----|------|------|
| **人类（晓力）** | 1 | 定义任务、跟进进展、最终验收 |
| **开发 Agent** | 多个 | 执行具体开发任务 |
| **Review Agent** | 1 | 代码审查、技术把关 |

### 任务分配规范

**晓力给 AI 分配任务时**，需要提供：

1. **任务描述** - 清晰、具体、可验收
2. **参考文档** - 相关文档路径
3. **验收标准** - 具体的验收条件
4. **执行限制** - 什么不能做

### 任务执行规范

**开发 Agent 执行任务时**：
1. 读取 `CONSTITUTION.md`
2. 读取 `docs/04-多AI协作流程.md`
3. 从 `tasks.md` 找到自己的任务
4. 开始执行任务
5. 完成后更新 `TODO.md` 状态

### 验收规范

**Review Agent 审查任务时**，必须检查：
- [ ] 代码质量（规范、注释、复杂度）
- [ ] 功能完整性（按需求实现）
- [ ] 测试覆盖（核心功能有测试）
- [ ] 文档更新（相关文档已更新）

审查通过后：
1. 更新 `TODO.md` 中任务状态为 ✅ 完成
2. 通知晓力验收

## 六、极简原则检查

每次实现功能时，必须问自己：

1. **这个功能必须吗？** - 不必须，删掉
2. **这个配置能去掉吗？** - 能去掉，去掉
3. **这个命令能简化吗？** - 能简化，简化
4. **这个参数能设默认值吗？** - 能，设默认值

**记住**：用户不会配置，用户只会点击和输入。

---

## 七、兼容性要求

### Markdown Memory 兼容

必须兼容现有的 Markdown Memory 系统：
- 能导入 Markdown 文件
- 能导出 Markdown 文件
- 保留原有的格式和结构

### 跨平台支持

必须支持以下平台：
- macOS (Intel & Apple Silicon)
- Windows (x64)
- Linux (可选)

### 同步空间兼容

必须能利用现有同步空间：
- 路径可配置
- 自动检测冲突
- 简单的合并策略
